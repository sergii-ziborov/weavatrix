# Weavatrix 0.2.5 development plan

Implementation status: all three tracks delivered. The complexity/hot-path stretch shipped only
after the precision and health regression gates were green.

## Release decision

`0.2.5` should ship two complete improvements, plus one gated stretch item:

1. **On-demand exact symbol inspection for TypeScript/JavaScript.** One bounded call returns the
   definition, exact reference evidence, grouped caller containers, graph blast radius, and small
   source excerpts. This combines the strongest missing LSP operation with the useful bounded source
   bundle seen in CodeGraph.
2. **Python call-resolution correctness.** Add receiver-aware method dispatch and safe wildcard-import
   resolution without weakening the existing alias and shadowing behavior.
3. **Delivered stretch: local hot-path review.** Expose and extend the complexity facts already
   persisted on graph nodes after the two precision tracks and their release gates are complete.

The release is not a five-feature grab bag. Generic CFG/data-flow/taint analysis is a separate engine,
not a patch-sized extension of the repository graph. Symbol rename is editor behavior. Both remain out
of scope for `0.2.5`.

## Why this scope

The current graph already wins on occurrence-level JS/TS calls, Git-aware change impact, endpoint
inventory, dead-symbol review, typed dependency evidence, duplicates, coverage, and repository-level
architecture. The comparison exposed two narrower correctness gaps:

- the global LSP overlay is intentionally capped (`32` symbols by default, `64` maximum), so a user
  cannot spend more precision budget on the one symbol currently under review;
- the Python adapter extracts only bare calls (`foo()`), so `receiver.run()` is not available to a
  receiver-aware resolver, and `from module import *` is not represented as a usable binding.

Complexity is not absent. `source-complexity.*` already records local cyclomatic complexity, loop
depth, allocation/copy signals, recursion, call counts, and local time/memory labels. Today those facts
are mostly graph payload and Hosted evidence-snapshot data, not a focused offline MCP workflow.

## Track A — exact symbol inspection and bounded context

### Public tool contract

Add an offline `source` capability tool named `inspect_symbol`.

Inputs:

- `label` (required): exact node ID or symbol label;
- `precision`: `auto | graph | lsp`, default `auto`;
- `max_references`: bounded `1..5000`, default `1000`;
- `max_containers`: bounded `1..50`, default `15`;
- `context_lines`: bounded `0..40`, default `8`;
- `timeout_ms`: bounded `1000..60000`, default `30000`;
- the common `output_format: text | json` switch.

Resolution must fail closed when a label identifies multiple symbols. The response returns candidate
IDs and asks the caller to select one; it must not silently run an exact query against the
best-connected same-name node.

The stable response contains:

- the resolved definition ID, kind, file, range, and bounded definition excerpt;
- evidence state: `EXACT`, `PARTIAL`, `GRAPH_ONLY`, `UNAVAILABLE`, or `OFF`;
- definition plus occurrence totals, unique files, and unique containing symbols;
- exact LSP locations grouped into logical caller containers, with per-container site counts;
- the existing graph dependents/blast radius, kept separate from exact occurrence evidence;
- bounded excerpts for the definition and highest-signal caller containers;
- completeness fields for reference, container, source, project, timeout, and cache caps;
- explicit static-only and exact-only deltas instead of silently replacing graph evidence.

An exact empty result becomes a zero-reference proof only under the existing complete-project and
revision/fingerprint checks. Otherwise the result is `PARTIAL` or `UNAVAILABLE`, never an exact dead
verdict.

### Implementation shape

1. Extract the reusable single-target work from `src/precision/lsp-overlay.js` into a precision query
   module. The global ranked overlay continues to call it for its bounded target set.
2. Keep `precision.json` as the global overlay. A point query must never overwrite it.
3. Store point-query results in a separate bounded cache keyed by graph revision, semantic-input
   fingerprint, provider contract, and exact symbol ID. Use atomic writes and an LRU/entry cap.
4. Reuse the bundled TypeScript language server, repository boundary checks, config/plugin refusal,
   source hashes, reference classification, deadline handling, and post-query fingerprint check.
5. Map each exact location to the innermost containing graph symbol. Group by container while retaining
   every occurrence in JSON mode.
6. Build excerpts through the existing repository-safe source reader; never accept absolute or
   traversal paths from graph/cache data.
7. Put the MCP adapter in a dedicated `tools-source.mjs` module so `tools-graph.mjs` remains a pure
   graph read layer. Add it to the hot-reload catalog explicitly.

### Acceptance fixtures

- `withConfigType` in `analytics`: definition plus `153` non-declaration references (`152` calls and
  one import), grouped without losing occurrence counts; no global-overlay safety-cap truncation.
- `normalizeEdgeAnalyticsTimeRangeUpdate`: exact zero references when the configured project fully
  covers the graph universe.
- same-name ambiguity returns candidates and performs no LSP request until an exact ID is supplied.
- type-only usage, runtime usage, re-export/import locations, Unicode positions, and same-line local
  shadowing retain their current classifications.
- a source edit, config edit, graph revision change, provider-contract change, or timeout cannot reuse
  stale exact evidence.
- the broad overlay remains unchanged before and after a point query.

Provisional performance gates on the comparison workstation: a cold one-symbol query completes in
`<=30s`; a valid cache hit completes in `<=1s`. Record measured time, provider state, caps, and cache
status in the response rather than hiding a miss or timeout.

## Track B — Python receiver dispatch and wildcard imports

### Symbol and binding model

1. Mark Python functions nested directly in a class as `method` symbols with `member_of`, visibility,
   and a structural class-to-method edge. Preserve module functions as functions.
2. Extend Python call extraction with attribute calls such as `receiver.run()` while retaining bare
   calls.
3. Build conservative receiver bindings from:
   - direct construction (`service = AlphaService()`), including imported aliases;
   - annotated parameters/locals (`service: AlphaService`);
   - `self`/`cls` within the owning class;
   - simple `self.field = AlphaService()` assignments when the binding is unambiguous.
4. Resolve a method call only when the receiver maps to one class and that class maps to one reachable
   method. Emit `RESOLVED`, not `EXACT_LSP`. Untyped, dynamic, monkey-patched, or ambiguous receivers
   produce no guessed call edge.
5. Keep lexical precedence explicit: local parameters/assignments and local declarations shadow
   explicit imports; explicit imports shadow wildcard bindings.

### Wildcard rules

1. Record `from module import *` as a wildcard binding to the resolved module file.
2. If a static `__all__` string list/tuple is available, only those names are eligible.
3. Without `__all__`, only public module-level names (not `_private`) are eligible.
4. A bare use resolves through wildcard imports only when exactly one eligible origin remains after
   lexical precedence. Multiple wildcard origins are reported as ambiguous and produce no call edge.
5. Re-export chains through package `__init__.py` are bounded and cycle-safe. Dynamic `__all__`,
   `__getattr__`, runtime import mutation, and star imports from unresolved external packages stay
   explicitly unresolved.

### Implementation boundary

Prefer a Python-specific pass-2 resolver hook over adding Python type rules to the generic name-only
`resolveCall` function. Shared pass-2 context may expose symbol tables, imports, enclosing symbols,
and edge emission, but Python binding/receiver semantics remain owned by `lang-python.js` or a small
Python resolver sibling.

Add a versioned Python-resolution marker and summary counters so benchmarks can distinguish supported,
resolved, ambiguous, and dynamic cases. Do not relabel parser-resolved edges as LSP-exact.

### Acceptance fixtures

- `AlphaService.run()` and `BetaService.run()` resolve to their own methods with no cross-edge.
- a constructor imported under an alias still binds the correct receiver class.
- annotated parameters resolve when their annotation has one internal class origin.
- `self.run()` resolves to the current class; an ambiguous inherited override is not guessed.
- a public function imported through `from package import *` resolves.
- static `__all__` excludes an otherwise public-looking name.
- local parameter/assignment shadowing suppresses a false imported or wildcard call.
- two wildcard modules exporting the same name produce no call edge and increment ambiguity evidence.
- existing Python, JS/TS, Go, Rust, Java, and C# golden relations do not regress.

## Track C — delivered local hot-path review

This track shipped after Tracks A and B passed their focused regression gates.

Add `hot_path_review` under the offline `health` capability. It ranks production symbols using the
already persisted local metrics plus graph fan-in/fan-out and test coverage. Its response must label
local syntax cost separately from graph risk.

Low-risk `0.2.5` extensions:

- allocations/copies inside loops;
- linear scans inside loops;
- recursion inside loops;
- configurable thresholds, path scope, production/test classification, and bounded `top_n`;
- source line/evidence pointers and explicit confidence.

Do not claim interprocedural Big-O in this release. Propagating loop degree through callees requires
call-site loop-depth metadata, SCC-safe recursion handling, and a separately versioned model. Keep
`transitiveLoopDepth`, guarded-recursion analysis, CFG unreachable branches, dead stores, and taint
source-to-sink evidence for a later milestone.

## Delivery order

1. Add contract fixtures for the two `analytics` symbols and the Python receiver/wildcard matrix.
2. Refactor the LSP query core without changing global-overlay output; run precision tests.
3. Implement `inspect_symbol`, its separate cache, grouping, source bundle, and JSON completeness.
4. Implement Python symbol ownership, lexical bindings, receiver resolution, and wildcard rules.
5. Run quick/golden/real benchmarks and compare relation/provenance deltas against `0.2.4`.
6. After the core tracks are green, implement and gate the bounded local `hot_path_review`.
7. Update README, bundled skill, MCPB catalog, version, third-party notices if needed, and release notes.

## Required tests and release gates

Unit/integration coverage:

- precision core, point-query cache, invalidation, ambiguity, caps, timeout, and no-reference safety;
- MCP text and stable JSON output, tool profiles, hot reload, repo retarget, and traversal refusal;
- Python AST forms, lexical shadowing, ambiguous negatives, ownership, provenance, and incremental/full
  graph parity;
- hot-path metric, classification, coverage-state and ranking tests for shipped Track C.

Release commands:

```sh
npm test
npm run benchmark
npm run benchmark:real:release
npm run build:mcpb
npm run verify:release
```

Release acceptance also requires:

- no unexplained golden or real-repository relation regression beyond the existing benchmark budget;
- no new runtime dependency unless measured evidence shows it is necessary;
- npm and MCPB behavior remain identical;
- no repository source, excerpt, absolute path, token, proxy, or environment value leaves the local
  process;
- graph-wide tools keep working when LSP precision is `OFF`, `UNAVAILABLE`, or `PARTIAL`;
- documentation states exactly which evidence is parser-resolved, graph-derived, or LSP-exact.

## Explicitly deferred

- generic CFG unreachable-branch and dead-store analysis;
- taint/data-flow source-to-sink evidence;
- interprocedural complexity propagation and guarded-recursion proofs;
- TypeScript merged type/value symbol separation;
- editor rename/refactoring;
- CodeGraphContext-style generic Cypher as a product surface.

The next decision after `0.2.5` should be evidence-based: compare exact-symbol latency/coverage,
Python fixture precision/recall, and user adoption of the bounded context response before choosing
between interprocedural hot paths and a private CFG/data-flow pilot.
