# Weavatrix 0.1.3 — signal quality

This release keeps Weavatrix's 23-tool surface intact while making structural findings materially
less noisy on real monorepos, Electron applications and Next.js workspaces.

## Highlights

### Runtime edges are no longer confused with type-only coupling

- JavaScript/TypeScript import and re-export edges carry type-only metadata.
- Runtime cycles and compile-time-only cycles are reported separately.
- `module_map` separates runtime dependencies from type-only module dependencies.
- `change_impact` labels compile-time-only paths while preserving a runtime path when one exists.
- `graph_diff` compares cycle membership and size without calling a smaller surviving SCC a new
  cycle. The first typed rebuild establishes a baseline instead of inventing a runtime/type delta.
- `god_nodes` ranks unique neighboring nodes, prioritizes runtime connectivity, and reports repeated
  references separately so a single heavily referenced import cannot look like 100 dependencies.

### Generated artifacts stop drowning out source findings

- Git repositories are scanned through the tracked plus non-ignored untracked file universe.
- Graph building and duplicate detection now agree on that boundary.
- Ignored packaged applications, Chromium licenses and stale generated graph entries are excluded
  from clone results.

### Better framework and workspace awareness

- Dependency checks use the nearest package manifest and nearest `tsconfig`/`jsconfig`, including
  nested path aliases and common config variants.
- Framework-owned runtime peers under Next.js, Vinext, Electron Vite and Cloudflare Vite are no
  longer reported as unused merely because the framework, rather than application source, loads
  them. Common `.mjs`/`.ts` tool configs are also included in usage evidence.
- Next.js App Router `route.ts`/`route.js` exports are recognized as endpoints and entry points.
- JSX namespace/member references such as `<ui.Button />` are retained in the graph.
- Class members are no longer treated as module exports, reducing unused-export noise.
- Optional `.weavatrix-deps.json` configuration can declare project-specific entry points and
  Python dependencies supplied by a managed runtime.

### Honest audit completeness

- OSV audit coverage is reported per repository as `OK`, `PARTIAL`, `NOT_CHECKED` or `ERROR`.
- A SHA-256 fingerprint ties an `OK` result to the exact supported pinned dependency set. Lockfile
  drift, legacy cache stamps, failed requests and malformed batch responses cannot produce a clean
  result.
- Missing coverage reports are shown as unavailable rather than as zero coverage, and the notice is
  not repeated for every impact node.

### More useful clone and architecture output

- Semantic duplicate mode ranks the closest clone candidate and the most divergent same-name pair
  independently instead of letting collisions crowd out clones.
- Common handler names such as `do_GET` and `do_POST` are not treated as meaningful semantic twins.
- Output now states explicitly that same-name divergence is not itself a duplicate or a bug; source
  review is required.
- Structural audit, module maps and impact reports use the typed-edge model consistently.

### Repository switching and hosted-sync contract

- `open_repo` still switches repositories in one explicit offline call. It now automatically builds
  a missing graph and upgrades pre-0.1.3 graphs; `build:false` remains a non-building probe and
  refuses legacy graphs.
- `sync_graph` emits allowlisted payload v2 with typed-edge metadata and a payload-version header.
  Source bodies and absolute repository paths are still excluded.

## Upgrade

```sh
npx -y weavatrix@0.1.3 <repoRoot>
```

Graphs built before 0.1.3 need one typed rebuild. Opening another repository normally performs the
upgrade automatically. If the server is already targeting an older active graph, call
`rebuild_graph` once before relying on runtime/type-only classifications or calling `sync_graph`.

Sync endpoint implementers must accept `syncPayloadV: 2` and typed link metadata. The local MCP
remains offline by default; `refresh_advisories` and `sync_graph` still require the explicitly
enabled `online` capability group and an individual tool call.

Requires Node.js 18 or newer. No tool was removed or moved to a different capability group.
