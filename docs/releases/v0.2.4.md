# Weavatrix 0.2.4

## Graph-mode correctness and semantic precision patch

This patch makes graph build mode part of the comparison and cross-repository refresh contract.

- Git-ref baselines inherit the active graph's `full`, `no-tests`, or `tests-only` mode.
- `graph_diff` refuses a previous snapshot whose mode differs from the current graph, preventing
  false node, edge, orphan and SCC removals across incomparable universes.
- Cross-repository HTTP tracing refreshes every selected graph in its persisted mode and reports
  `buildMode` beside the refresh status.
- `open_repo` preserves an existing graph mode unless another one is requested explicitly. A
  `build:false` probe fails closed on a mode mismatch, while successful targeting reports canonical
  repository path, graph path and build mode.
- `graph_stats` now exposes graph path and mode so precision-sensitive work can verify its active
  target after an MCP reconnect.
- `module_map` excludes classified non-product surfaces by default. Set
  `include_non_product:true` to include tests, fixtures, benchmarks, generated output and docs.
- A bounded, revision-matched precision overlay now launches Weavatrix's own bundled
  `typescript-language-server` 4.4.1 with TypeScript 5.9.3 for TypeScript and JavaScript
  `textDocument/references`. Resolution is anchored to the Weavatrix installation: it does not run
  repository scripts, invoke `npx`, download a server, or send source over the network.
- Only references returned by that semantic provider become `EXACT_LSP`. Successful exact
  zero-reference queries can strengthen internal dead-code evidence; partial, failed, capped, stale,
  and unsupported results remain explicit rather than being promoted.
- Before provider startup, applicable repo-contained TypeScript/JavaScript config chains are parsed
  without executing them. Configured language-service plugins and unresolved/outside inputs are
  refused. A bounded semantic-input fingerprint invalidates ignored/configured changes before cache
  reuse and is checked again after the provider run.
- `graph_stats` reports provider, version, bounded coverage and `COMPLETE`, `PARTIAL`, `UNAVAILABLE`,
  or explicitly disabled `OFF` state. Java and Rust precision providers are not bundled in 0.2.4;
  their edges never become `EXACT_LSP` even when a mixed repository has a complete
  TypeScript/JavaScript overlay, while their parser graph remains usable.
- New graphs enable the bounded JS/TS provider by default. `WEAVATRIX_PRECISION=off` disables it
  before the first npm/server build, and the MCPB installer exposes the same `lsp` / `off` choice.
- Broad `query_graph` bootstrap and tool-execution ranking now prioritizes production executables
  and graph-declared entry points over site, documentation, benchmark and fixture matches. It remains an
  orientation tool; exact `seed_files` are deterministic when the entry point is already known.
- The bundled skill routes exact TypeScript/JavaScript reference and dead-code work through this
  built-in evidence instead of recommending another product when `EXACT_LSP` is zero.

The two semantic-provider packages are exact, production runtime dependencies so npm and MCPB
installs behave identically. Release verification pins both versions and the MCPB build fails if its
staged language-server runtime, Apache-2.0 licenses, or TypeScript third-party notice is missing.
